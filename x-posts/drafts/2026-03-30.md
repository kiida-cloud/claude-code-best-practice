---

Claude Codeの最大の生産性向上策はgit worktreeによる並列実行だ。

- `git worktree add ../feature-x feature-x` で即座に分離環境を作る
- 3〜5個のworktreeを立ち上げ、各Claudeセッションで別タスクを同時進行
- シェルエイリアス `2a`, `2b`, `2c` でワンキーで切り替える人も多い
- 「分析専用」worktreeをログ確認・BigQueryクエリ専用にする使い方も有効

## 2026-03-31 (火) — Prompting

カテゴリ: Prompting
フォーマット: thread
ソース: tips/claude-boris-10-tips-01-feb-26.md (Tip 2)

---

ツイート1:
複雑なタスクをいきなりClaudeに渡していないか？

ツイート2:
plan modeで徹底的に計画してから実装させると、ほぼ1発で仕上がる。

- Shift+Tab でplan modeに切り替え
- 1人がプランを書き、別のClaudeをstaff engineerとしてレビューさせる使い方も有効
- 途中でおかしくなったらすぐplan modeに戻って再計画する。そのまま押し進めない
- 実装だけでなく、検証ステップにもplan modeを使う

ツイート3:
「実装に入る前の計画に全エネルギーを注ぐ」がClaude Codeチームの共通認識。計画が固まれば、実装はClaudeに任せられる。

---

Claudeがミスをしたとき、最後に「CLAUDE.mdを更新して同じミスをしないようにして」と伝えるだけでいい。

Claudeは自分向けのルールを書くのが異様に得意だ。あとは定期的に削ぎ落とし、ミス率が実際に下がるまで繰り返す。タスクごとにnotesディレクトリを管理させ、CLAUDE.mdからそこを参照させる方法も効果的。

## 2026-04-03 (金) — Subagents

カテゴリ: Subagents
フォーマット: thread
ソース: tips/claude-boris-2-tips-10-mar-26.md

---

ツイート1:
同じモデルを使っても、コードを書いたコンテキストよりも別のコンテキストの方がバグを見つけやすい。なぜか？

ツイート2:
これがtest time computeの本質。トークンを多く使うほど精度が上がり、さらに複数の独立したコンテキストウィンドウを使うと相乗効果がある。

コードを書いたエージェントとレビューするエージェントを分ける理由がここにある。自分が書いたコードより、同僚がレビューした方がバグを見つけやすいのと同じ構造だ。

ツイート3:
実践方法はシンプル。「use subagents」と付けるだけでClaudeが問題に複数コンテキストを割り当てる。Code Reviewでも同じ仕組みが動いている。

## 2026-04-05 (日) — Git Workflow

カテゴリ: Git Workflow
フォーマット: single
ソース: tips/claude-boris-2-tips-25-mar-26.md

---

Boris Chernyが1日141PRを出したときの中央値は118行。常にsquash merge。

- 1PR = 1コミットで歴史がクリーンになり、revertも git bisect も楽になる
- p90でも498行以下。大きなPRは例外であり、標準ではない
- 小さいPRはマージコンフリクトを減らし、レビューコストを下げる

AI支援の高速開発では「fix lint」「try this」コミットはノイズ。squashが実用的な選択肢になる。
