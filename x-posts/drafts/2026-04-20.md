## 2026-04-20 (月) — Productivity / Workflow

カテゴリ: Productivity
フォーマット: single
ソース: best-practice/claude-skills.md (loopスキル), best-practice/claude-settings.md

---

/loop スキルを使うと Claude Code で定期タスクを自動実行できる。

- /loop 5m /foo — 5分ごとに /foo スキルを実行
- デフォルトは10分間隔、最大3日間継続
- /schedule はクラウドで実行 — マシンをオフにしても動き続ける

CI待ちの間に /loop 30m /deploy-check を仕込んでおくと、デプロイ完了を自動で拾える。

## 2026-04-21 (火) — Prompting

カテゴリ: Prompting
フォーマット: thread
ソース: tips/claude-boris-10-tips-01-feb-26.md (Tip 6)

---

ツイート1:
Claude に「このコードをレビューして」と頼むより、はるかに効果的な聞き方がある。

ツイート2:
Claude をレビュアーの立場に置くプロンプト:

- "Grill me on these changes and don't make a PR until I pass your test."
  — Claude が審査員になり、自分が説明できるまで PR を作らない
- "Prove to me this works"
  — main と feature ブランチの差分を比較させ、動作を証明させる
- "Knowing everything you know now, scrap this and implement the elegant solution."
  — 実装が微妙だったとき、一度捨てて最善策を出させる

ツイート3:
「評価者」の立場に置くだけで Claude の出力の質が変わる。レビューだけでなく、設計や技術調査でも同じ効果がある。

## 2026-04-22 (水) — Memory / Context

カテゴリ: Memory
フォーマット: single
ソース: best-practice/claude-settings.md (CLAUDE_AUTOCOMPACT_PCT_OVERRIDE, spinnerTipsOverride)

---

コンテキスト使用率50%で /compact を手動実行するのが最良のタイミングだ。

- 自動 auto-compact はデフォルト95%で走る — 直前の作業内容が消えることがある
- 50%で叩くと重要な判断履歴を保持しながら圧縮できる
- ステータスラインに使用率を表示しておくと見落としが減る(/statusline で設定)

CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50 を env フィールドに設定すると自動 compact のタイミングも変えられる。

## 2026-04-23 (木) — Skills

カテゴリ: Skills
フォーマット: single
ソース: tips/claude-thariq-tips-17-mar-26.md (Tip 6)

---

Skill の description フィールドはユーザー向けの説明文ではない。

Claude Code がセッション開始時に全スキルの description 一覧をスキャンし、「このリクエストにどのスキルを使うか」を判断する。つまりここはモデルへのトリガー条件を書く場所だ。

悪い例: "BigQuery クエリを実行するスキル"
良い例: "ユーザーがデータ分析、BigQuery クエリ、または指標の調査について聞いてきたときに使用"

## 2026-04-24 (金) — Subagents

カテゴリ: Subagents
フォーマット: thread
ソース: reports/claude-agent-command-skill.md

---

ツイート1:
Claude Code の Agent / Command / Skill — 3つの違いを一言で言えるか？

それぞれ「誰が起動するか」と「コンテキストを分離するか」が根本的に違う。

ツイート2:
使い分けの基準:

- Agent: 複数ステップの自律タスク、コンテキストを汚したくない場合
  → /メニューには出ない。Claude が description を読んで自動起動する
- Command: ユーザーが明示的に起動するワークフロー
  → /xxx で呼ぶ。自動起動なし
- Skill: 再利用可能な手順、Claude に自動検出させたい場合
  → /メニューに出る。description でトリガー条件を指定

ツイート3:
3つを組み合わせた Command → Agent → Skill のパターンが最も強力。

Command がエントリーポイント、Agent が独立コンテキストで自律実行、Agent 内の Skill がドメイン知識を提供する。チームの繰り返し作業はこの構造に落とし込める。

## 2026-04-25 (土) — Hooks

カテゴリ: Hooks
フォーマット: single
ソース: tips/claude-boris-13-tips-03-jan-26.md (Tip 9), best-practice/claude-settings.md

---

PreToolUse フックで git commit 前に独自バリデーションを強制できる。

settings.json への追加例:

"PreToolUse": [{
  "matcher": "Bash(git commit*)",
  "hooks": [{
    "type": "command",
    "command": "bun run lint && bun run typecheck"
  }]
}]

lint や型チェックが失敗するとコミットが中断される。--no-verify を回避策に使わせずに済む。.claude/settings.json にコミットすればチーム全員に適用される。

## 2026-04-26 (日) — MCP

カテゴリ: MCP
フォーマット: single
ソース: tips/claude-boris-10-tips-01-feb-26.md (Tip 5), tips/claude-boris-13-tips-03-jan-26.md (Tip 11)

---

Slack MCP を入れると、バグ報告スレッドをそのまま Claude に渡せる。

手順:
- .mcp.json に Slack MCP サーバーを追加
- バグのある Slack スレッドを Claude に貼り付ける
- "fix" と言うだけ

Claude がスレッドのコンテキストを読んで該当コードを修正する。チケットツールとのコンテキストスイッチがゼロになる。.mcp.json は git 管理してチームで共有できる。
